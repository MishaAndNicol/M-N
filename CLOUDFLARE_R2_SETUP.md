# Cloudflare R2 — настройка хранилища видео

Watch Room больше не зависит от Google Drive. Видео хранятся в Cloudflare R2 и
отдаются как обычные публичные HTTPS-файлы, а `<video>` на странице
проигрывает их напрямую — сервер не нужен, GitHub Pages это переживёт без
проблем.

## 1. Создать R2 Bucket

1. Зайдите в [dash.cloudflare.com](https://dash.cloudflare.com) → в левом меню
   выберите **R2 Object Storage**.
   (Если R2 включаете впервые — Cloudflare попросит привязать способ оплаты;
   есть бесплатный тариф — 10 ГБ хранилища и щедрый лимит исходящего трафика
   без платы за egress, этого более чем достаточно для личного проекта.)
2. **Create bucket**.
   - Bucket name: например `two-story-films` (любое, латиницей, без пробелов).
   - Location: по умолчанию (Automatic) — нормально.
   - Остальное можно оставить как есть → **Create bucket**.

## 2. Включить публичный доступ

По умолчанию бакет приватный. Нужно включить публичное чтение — самый простой
способ для статического сайта:

1. Откройте созданный бакет → вкладка **Settings**.
2. Раздел **Public access** → **R2.dev subdomain** → **Enable**.
3. Cloudflare выдаст публичный домен вида:
   ```
   https://pub-XXXXXXXXXXXXXXXXXXXXXXXXXXXX.r2.dev
   ```
   Это и есть базовый URL, на котором будут жить видео. Скопируйте его.

   Альтернатива (не обязательно, но приятнее): в том же разделе **Custom
   Domains** можно привязать свой поддомен (например `videos.вашдомен.ru`),
   если домен уже подключён к Cloudflare — тогда используйте его вместо
   `r2.dev`-адреса.

**Важно про приватность.** Публичный бакет означает, что любой, кто узнает
прямую ссылку на файл, сможет его открыть — просто по знанию URL, без пароля.
Для семейного/личного watch-room-проекта это обычно приемлемо (ссылки никто не
публикует), но не используйте это для действительно приватного контента.
Список файлов бакета публично не просматривается — открыть можно только файл
по точному URL, если только вы отдельно не включите bucket listing.

## 3. Настроить CORS (обычно не требуется, но на всякий случай)

`<video src="...">` не требует CORS-заголовков для обычного воспроизведения —
браузер разрешает кросс-доменное видео без них (CORS нужен только если читать
пиксели/кадры через `<canvas>`, чего в проекте нет). Так что этот шаг можно
пропустить.

Если позже понадобится (например, будете использовать Media Source Extensions
или превью-кадры через canvas), в бакете: **Settings → CORS Policy → Add CORS
policy**:

```json
[
  {
    "AllowedOrigins": ["https://ВАШ_ЛОГИН.github.io"],
    "AllowedMethods": ["GET", "HEAD"],
    "AllowedHeaders": ["*"],
    "MaxAgeSeconds": 3600
  }
]
```

## 4. Загрузить видео

Вариант A — через Dashboard (проще всего для пары файлов):
1. Откройте бакет → **Objects** → **Upload**.
2. Перетащите видеофайлы (`.mp4` рекомендуется — универсальная поддержка во
   всех браузерах; `.webm` тоже работает, но hevc/`.mov` не проигрываются в
   части браузеров).
3. После загрузки у каждого файла есть свой публичный URL:
   ```
   https://pub-XXXXXXXXXXXXXXXXXXXXXXXXXXXX.r2.dev/имя-файла.mp4
   ```

Вариант B — через `rclone` или AWS CLI (для больших файлов/много файлов
удобнее, R2 совместим с S3 API):
```bash
# один раз настроить профиль (Access Key создаётся в
# R2 → Manage R2 API Tokens → Create API Token)
aws configure --profile r2
# Endpoint URL для всех команд:
# https://<account_id>.r2.cloudflarestorage.com

aws s3 cp ./episode-01.mp4 s3://two-story-films/episode-01.mp4 \
  --profile r2 --endpoint-url https://<account_id>.r2.cloudflarestorage.com
```

Имена файлов используйте латиницей, без пробелов (`episode-01.mp4`, а не
`Серия 1.mp4`) — так URL остаётся простым и без проблем с кодировкой.

## 5. Подключить к сайту

В приложении это уже готово к работе. Есть два равноценных способа указывать
видео в Watch Room — можно пользоваться и без каких-либо новых переменных
окружения:

**Способ 1 — просто вставлять полный URL (ничего дополнительно настраивать не
надо):**
В поле "R2 public video URL" на `/watch` вставляйте прямо
`https://pub-XXXX.r2.dev/episode-01.mp4`.

**Способ 2 — один раз задать базовый URL, дальше вставлять только имя файла:**
Добавьте переменную окружения `NEXT_PUBLIC_R2_PUBLIC_BASE_URL` (см. ниже) со
значением `https://pub-XXXX.r2.dev`, и тогда в Watch Room можно вставлять
просто `episode-01.mp4`.

## 6. Переменные окружения / GitHub Secrets

Единственное изменение относительно текущей настройки: переменная
`NEXT_PUBLIC_GOOGLE_DRIVE_API_KEY` больше не используется — её можно удалить.
Вместо неё **опционально** добавляется:

| Переменная | Обязательна? | Значение |
|---|---|---|
| `NEXT_PUBLIC_R2_PUBLIC_BASE_URL` | Нет (только для способа 2 выше) | `https://pub-XXXXXXXXXXXX.r2.dev` (без слэша в конце) |

Локально: добавьте строку в `.env.local` (скопируйте пример из
`.env.example`).

На GitHub: **Settings → Secrets and variables → Actions → New repository
secret**, имя `NEXT_PUBLIC_R2_PUBLIC_BASE_URL`, значение — ваш публичный
R2-домен. Если старый секрет `NEXT_PUBLIC_GOOGLE_DRIVE_API_KEY` там есть — его
можно удалить, он больше не читается сборкой (workflow уже не ссылается на
него).

Все остальные секреты (Firebase, `NEXT_PUBLIC_ADMIN_EMAIL`) не менялись.

## 7. Деплой

Ничего нового делать не нужно — тот же процесс, что и раньше:
```bash
git add .
git commit -m "Switch video source from Google Drive to Cloudflare R2"
git push
```
GitHub Actions соберёт статический сайт (`npm run build`, `output: "export"`)
и выложит его на GitHub Pages, как и раньше — сервер по-прежнему не нужен.

## Как это работает под капотом (коротко)

- Раньше: `src/lib/drive.ts` собирал URL вида
  `https://www.googleapis.com/drive/v3/files/{id}?alt=media&key=...`, что
  требовало API-ключ Google и публичный доступ "Anyone with the link".
- Теперь: `src/lib/r2.ts` просто принимает URL, который уже выдал Cloudflare —
  ключ не нужен, потому что доступ на чтение публичный сразу на уровне
  бакета/объекта.
- `<video src="...">` в `watch-room.tsx` работает так же, как и раньше:
  R2 поддерживает HTTP Range-запросы "из коробки", поэтому перемотка и
  прогрессивная загрузка работают без какого-либо прокси-сервера.
- Синхронизация play/pause/seek/currentTime через Firestore (`watchRoom/room`)
  не менялась вообще — она никогда не зависела от источника видео.
